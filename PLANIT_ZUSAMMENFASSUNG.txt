PLANIT - ZUSAMMENFASSUNG FÜR NEUEN CHAT
========================================
Datum: 13. April 2026

WAS WIR BAUEN:
--------------
PlanIt ist eine eigenständige Mitarbeiter-App für Schichtplanung.
Komplett separate App, nutzt gleiche Supabase-Datenbank wie GastroHub.
Deployed auf Cloudflare Pages: planit-886.pages.dev

TECHNISCHER STACK:
------------------
- Vanilla HTML/CSS/JS (keine Frameworks)
- Supabase als Backend (gleiche DB wie GastroHub)
- Supabase URL: https://ulazvcrpasnhufzlywxf.supabase.co
- GitHub: github.com/ArtsemKozel/planit
- Cloudflare Pages: planit-886.pages.dev

LOGIN-SYSTEM:
-------------
- Admin: Email + Passwort (Supabase Auth)
- Mitarbeiter: Login-Kürzel + PIN
- Supabase-Client heißt "db"
- Mitarbeiter-Session in localStorage
- index.html: Admin-Tab initial aktiv (nicht Mitarbeiter)
- register.html: Registrierung neuer Restaurants (Supabase Auth + planit_restaurants)
  → Validierung: Restaurant ≥3 Zeichen, Email-Regex, PW ≥8/Großbuchstabe/Zahl/Sonderzeichen
  → Nach Registrierung: Erfolgsmeldung statt Redirect
  → window.register als explizit globale Funktion, Credentials inline

DATENBANK-TABELLEN:
-------------------
1. employees_planit (id uuid, user_id, name, login_code, password_hash, department,
   departments (kommasepariert, Multi-Abteilung), is_active, birthdate,
   vacation_days_per_year, is_apprentice, start_date, hours_per_vacation_day,
   can_do_inventory boolean,
   hygiene_erste date, hygiene_letzte date, hygiene_gueltig_monate int)

2. shifts (id, user_id, employee_id, shift_date, start_time, end_time, break_minutes,
   notes, is_open, open_note, department, handover_requested, handover_employee_id,
   actual_start_time, actual_end_time, actual_break_minutes)
   → actual_* Felder: tatsächliche Zeiten, werden für Stundenkonto und tip_hours genutzt

3. vacation_requests (id, user_id, employee_id, start_date, end_date, reason, status,
   rejection_reason, reviewed_at, reviewed_by, deducted_days, type, payout_month, pdf_url)

4. availability, shift_swaps, shift_handovers, payroll_documents (wie vorher)

5. approved_hours (id, user_id, employee_id, month, approved_minutes)

6. actual_hours (id, user_id, employee_id, month, actual_minutes, carry_over_minutes)
   → actual_minutes nicht manuell; carry_over_minutes bleibt

7. open_shift_requests, shift_templates, sick_leaves, employment_phases (wie vorher)

8. tasks (id, user_id, title, created_at, is_archived)
9. task_steps (id, user_id, task_id, title, is_done, position)
10. task_templates, task_template_steps, notes (wie vorher)

11. tip_config (id, user_id, mode, show_to_employees, fixed_results jsonb)
12. tip_departments (id, user_id, department, percentage, pool_department)
    → pool_department: Falls gesetzt, wird diese Abteilung über den Tagesschnitt
      in den Küche-Pool eingerechnet (z.B. Bäckerei → Küche)
    → fixed_hours_per_month wurde entfernt; Stunden kommen jetzt immer aus shifts
13. tip_entries (id, user_id, entry_date, amount_card, amount_cash)
    → Unique Constraint: (user_id, entry_date)
    → Alle Tage des Monats werden beim Laden automatisch angelegt (upsert)
14. tip_hours (id, user_id, employee_id, work_date, minutes, start_time, end_time,
    break_minutes, department)
    → Unique Constraint: (user_id, employee_id, work_date, department)
    → Wird automatisch aus shifts befüllt (Sync beim Laden + nach Schicht-Änderungen)
    → department aus Schicht-Department (für Multi-Abteilung)
15. tip_results (id, user_id, employee_id, month, amount_card, amount_cash)

16. planit_suppliers, planit_inventory_items, planit_inventory_entries,
    planit_inventory_groups, planit_inventory_submissions (wie vorher)

17. planit_departments (id, user_id, name)
    → Neue Tabelle für Abteilungsverwaltung
    → Ersetzt hardcodierte Abteilungsnamen (Service, Küche, Bar)
    → Verwaltet im Team-Tab unter "ABTEILUNGEN"

18. planit_restaurants (id, user_id, name, hygiene_link_erst, hygiene_link_erneuerung, ...)
    → Wird bei Registrierung angelegt
    → hygiene_link_erst / hygiene_link_erneuerung: Links für Belehrungsformulare

19. planit_terminations (id, user_id, employee_id, requested_date, approved_date,
    status, reason, pdf_url, street, zip, city)
    → status: 'pending' | 'approved' | 'rejected'
    → approved_date: wird als cutoffDate im Urlaubskonto verwendet

SUPABASE STORAGE:
-----------------
- Bucket: vacation-pdfs (nicht public)
- Bucket: termination-pdfs (public) — Kündigungs-PDFs
